> **定位：** 这是"把项目做到最好"的全面改进方案，不是最小可改补丁。核心目标是让创新点突出、竞争力足够强、UI 前端达到一等奖展示水准。**本阶段只产出方案文档，不修改任何代码。**
>
> **制定日期：** 2026-07-04 Asia/Shanghai
> **前置依据：** 第 1、2 轮健康体检报告（`docs/plans/2026-07-04-health-check-and-improvement-plan.md` 与 round2 版本）
> **赛事背景：** OSC2026 MoonBit 开源生态赛（评分维度：完成度 / 生态贡献 / 工程质量 / 展示）+ 区块链技术课程大项目

## 一、战略升级：从"验证库"到"可信证据工作台"

### 1.1 当前定位的瓶颈

当前 MoonEvidence 的自我定位是"chain-agnostic verification core"——一个**库**。这个定位在工程上是对的（纯核心、零 IO、三后端可移植），但在**展示层面是灾难性的**：

- 评委打开仓库，看到的是 CLI 文本输出和一张"OK/FAILED"的网页 demo。
- 项目底层有 Merkle 树、版本链、Ed25519 签名、审计日志、内容寻址存储、Merkle 包含证明——**这些全是天生适合可视化的结构，但前端一个都没暴露**。
- README 强调"不是区块链应用"——这虽然诚实，但**主动放弃了区块链课程大项目最有展示力的叙事钩子**。

### 1.2 升级后的定位

**MoonEvidence = 可信证据工作台（Trust Workbench）**

一句话：**让任何人在浏览器里"看见"密码学如何保护文件完整性——从字节到 Merkle 根到签名链，全程可视、可交互、可篡改实验。**

这个定位同时满足三个身份：
- **MoonBit 生态项目**：纯 MoonBit 核心编译到 Wasm/JS，浏览器本地验证，文件不离开电脑——展示 MoonBit 的跨后端能力。
- **区块链课程大项目**：Merkle 树、哈希链、数字签名是区块链的密码学地基，工作台把这些地基"可视化教学化"。
- **OSC2026 竞赛作品**：从"库"升级为"工作台"，展示维度直接拉满，且有真实使用场景（数据集归档、AI 输出审计、数字版权打包）。

### 1.3 为什么这个定位能冲击一等奖

对照赛方四维度：

| 维度 | 当前状态 | 升级后 |
| --- | --- | --- |
| 完成度 | 12 包 / 8368 行 / 240 测试 / 全绿 | 持续叠加可视化与交互层，完成度从"库完成"到"工作台完成" |
| 生态贡献 | Mooncakes 包 + 浏览器适配器 | 工作台是 MoonBit 在"可信基础设施"赛道的标杆 demo |
| 工程质量 | CI 全覆盖、契约冻结、自动防漂移（第 2 轮计划） | 叠加可视化测试、视觉回归、可访问性达标 |
| **展示** | **单页 OK/FAILED 表格——最弱项** | **交互式信任链可视化平台——最强项** |

## 二、创新点矩阵（6 个突出创新点）

> 每个创新点都满足：① 真实利用底层已有能力（不空造功能）；② 评委 30 秒内能"看懂"；③ 有差异化，MoonBit 生态无同类。

### 创新点 1：Merkle 树交互式可视化与"篡改实验室"

**底层能力：** `@merkle.compute_root` / `@merkle.compute_proof` / `@merkle.verify_inclusion`（已实现，CLI 已暴露 prove/check-proof）

**前端呈现：**
- 证据包验证后，自动渲染整棵 Merkle 树（SVG/Canvas），叶子节点 = 文件，内部节点 = 哈希。
- **篡改实验室模式**：用户在网页上点任一叶子节点，修改对应文件的任一字节，实时看到：
  - 该叶子节点变红（digest 不匹配）
  - 沿路径向上的所有祖先节点依次变红（哈希链断裂）
  - 根节点变红，弹出 "E3003 merkle root mismatch"
- 同时高亮"原证明路径"vs"篡改后路径"，直观展示 Merkle 证明的局部可验证性。
- **教育价值**：这是区块链课程最核心的"Merkle 树为什么能防篡改"的可视化证明，比任何 PPT 都直观。

**差异化：** 大多数 Merkle 可视化工具是静态图。MoonEvidence 的版本是**真实密码学驱动**——改一个字节，真实的 SHA-256 重算，真实的根哈希对比，不是动画模拟。

### 创新点 2：版本链时间轴 + DAG 演进可视化

**底层能力：** `@verify.verify_chain`（线性链已实现；ROADMAP 列了 DAG 为未来方向）

**前端呈现：**
- 线性版本链渲染为时间轴（oldest → newest），每个节点显示 version_id / parent / 时间戳。
- 点击任一节点，侧栏显示该版本的 manifest 摘要。
- **断链可视化**：构造一个 parent 缺失的版本链，时间轴上对应位置显示"断裂"图标 + E4001 错误标注。
- **未来 DAG 扩展占位**：时间轴布局预留 DAG 模式（分叉/合并），即便 MVP 不实现 DAG，UI 也展示"项目有架构延展性"。

**差异化：** 把"版本链验证"从一行 CLI 输出变成可探索的时间轴，评委能直观理解"为什么需要版本链"。

### 创新点 3：Ed25519 签名验证流程动画

**底层能力：** `@crypto.ed25519_sign` / `ed25519_verify`（纯 MoonBit 实现，~800 行，RFC 8032 KAT 通过）

**前端呈现：**
- 审计日志的每条 entry 可签名。签名后，工作台展示"签名验证流程动画"：
  - Step 1：消息（canonical JSON）→ 哈希
  - Step 2：签名 (R, S) + 公钥 A
  - Step 3：计算 `[8 * S] * R` 与 `[8] * A` 的关系
  - Step 4：验证通过/失败
- 动画基于真实的纯 MoonBit Ed25519 实现——**这是 MoonBit 生态唯一的纯语言实现 Ed25519**，本身就是生态贡献亮点。
- 提供签名生成器：用户输入消息，生成密钥对 + 签名，可下载。

**差异化：** 把"我们实现了 Ed25519"从一行 README 变成可操作的可视化，且突出"纯 MoonBit、无外部依赖"的生态价值。

### 创新点 4：审计日志哈希链实时可视化

**底层能力：** `@audit.AuditLog`（hash-chained append-only，已实现 verify_chain / sign_last）

**前端呈现：**
- 审计日志渲染为垂直哈希链：每条 entry 显示 action / target / hash / prev_hash / 签名状态。
- 用户可点击"追加新记录"，实时看到新 entry 的 hash 如何链接到上一条。
- **篡改演示**：用户点击某条历史 entry 修改 target，整条链从该点之后全部变红（hash 链断裂），对应 `verify_chain` 返回 false。
- 签名状态用锁图标标识（已签名/未签名），点击可触发验签动画（联动创新点 3）。

**差异化：** 审计日志通常是后台日志文件。MoonEvidence 把它变成可交互的"信任链仪表盘"。

### 创新点 5：从"验证器"到"工作台"——在线创建 + 包含证明

**底层能力：** `@create.create_manifest` / `@store.ObjectStore` / `@merkle.compute_proof`（全已实现）

**前端呈现：**
- **创建模式**：用户拖拽一组文件到网页，工作台实时计算每个文件摘要、组装 manifest、计算 Merkle 根，生成可下载的证据包 zip。全程浏览器本地，文件不上传。
- **包含证明模式**：选任一文件，生成 Merkle 包含证明，可视化证明路径（联动创新点 1），并支持"导出证明 + 验证证明"闭环。
- **去重热力图**：拖入多个证据包，用内容寻址存储展示 SHA-256 去重效果——相同文件只存一份，热力图显示重复率。

**差异化：** 从"只能验"变成"能建能验能证"，且全部在浏览器本地完成——这是 MoonBit 跨后端能力的最强展示。

### 创新点 6：暗色模式 + 可访问性 + PWA 离线

**底层能力：** 验证核心纯前端，天然适合 PWA

**前端呈现：**
- 完整暗色模式（与 IDE 主题联动），所有可视化在双模式下都清晰。
- WCAG AA 可访问性：键盘导航、屏幕阅读器标签、对比度达标。
- PWA：可安装、可离线。验证不依赖网络——这正是"chain-agnostic verification core"定位的最佳体现。

**差异化：** 工程质量细节，体现"这不是 demo，是产品"。

## 三、UI/前端全面重设计

### 3.1 信息架构（从单页到多视图工作台）

当前：单页 → 选目录 → 显示结果。

重设计为**多视图工作台**，顶部导航切换：

```
Trust Workbench
├── 验证 (Verify)          ← 当前 demo 升级版，默认视图
├── 创建 (Create)          ← 创新点 5：拖拽建包
├── 证明 (Prove)           ← 创新点 5：包含证明生成与验证
├── 审计 (Audit)           ← 创新点 4：哈希链可视化
├── 签名 (Sign)            ← 创新点 3：Ed25519 签名器
├── 篡改实验室 (Tamper Lab) ← 创新点 1：Merkle 篡改实验
└── 基准 (Bench)           ← 性能可视化（已有 _bench_data.csv）
```

每个视图共享同一套"证据包状态"——用户在一个视图加载的包，切换视图后仍可用，避免重复加载。

### 3.2 视觉系统升级

当前：GitHub 风卡片，单一紫色 accent，无层次。

重设计：

- **配色**：保留紫色（#6e40c9）作为主品牌色（与 MoonBit 月亮主题呼应），但建立完整的语义色系统：
  - 信任绿（OK）/ 警示红（FAIL）/ 注意黄（WARN）/ 信息蓝（INFO）
  - 暗色模式独立调色，不是简单反色
- **字体**：标题用几何无衬线（Inter / 思源黑体），代码用等宽（JetBrains Mono），数据用 tabular-nums
- **层次**：卡片 → 区块 → 行 三级，用阴影 + 边框 + 间距区分，而非全平
- **动效**：验证过程有"流水线"动画（parse → canonicalize → digest → merkle → chain 逐步点亮），篡改时节点有"断裂"动效，但遵守 `prefers-reduced-motion`
- **品牌**：设计一个"月亮 + 盾牌 + 哈希链"的组合 logo，用于 favicon / 顶栏 / 加载状态

### 3.3 交互设计要点

- **拖拽优先**：创建/验证视图都支持拖拽目录或文件到页面，比"选择目录"按钮更直观
- **实时反馈**：拖入后立即显示文件列表 + 实时摘要计算进度条
- **错误可定位**：findings 表格的每行可点击，点击后高亮对应的可视化节点（如点 E2003 行 → Merkle 树对应叶子高亮）
- **可分享状态**：验证结果 URL 可复制（state 编码到 hash），他人打开同一 URL 看到同一结果（注：文件本身不分享，只分享 manifest + findings）
- **键盘流**：Cmd/Ctrl+V 粘贴 manifest、Cmd/Ctrl+O 打开目录、Tab 切换视图

### 3.4 可视化模块技术选型

| 可视化 | 技术 | 理由 |
| --- | --- | --- |
| Merkle 树 | 自绘 SVG（递归布局） | 树结构清晰，可缩放可点选，无需重型图库 |
| 版本链时间轴 | SVG + CSS | 线性结构简单，DAG 预留用 dagre 布局 |
| Ed25519 动画 | SVG + CSS keyframes | 流程动画，无需 3D |
| 审计哈希链 | HTML 卡片 + SVG 连线 | 垂直链，卡片即可 |
| 去重热力图 | Canvas | 大量文件时性能好 |
| 性能图表 | Chart.js（轻量） | 已有 csv 数据，折线/柱状 |

**不引入 React/Vue 等框架**——保持 demo 的"零构建"特性，用原生 ES Modules + 轻量状态管理，与 MoonBit 编译产物直接集成。这本身也是工程亮点（无框架依赖的可维护前端）。

## 四、后端 API 扩展需求

当前 `@api.verify_evidence` 是单入口。工作台需要更多能力暴露到 JS 边界。**所有新增 API 都遵循现有契约：string-in/string-out，无 MoonBit 类型跨界。**

| 视图 | 需要的新 API | 底层已有 | 需新增包装 |
| --- | --- | --- | --- |
| Create | `create_evidence_pack(request_json)` | `@create.create_manifest` | 是，包装为 JSON API |
| Prove | `generate_proof(request_json)` / `verify_proof(request_json)` | `@merkle.compute_proof` / `verify_inclusion` | 是，包装为 JSON API |
| Audit | `audit_append(request_json)` / `audit_verify(request_json)` / `audit_sign(request_json)` | `@audit.*` | 是，包装为 JSON API |
| Sign | `ed25519_keypair(request_json)` / `ed25519_sign(request_json)` / `ed25519_verify(request_json)` | `@crypto.*` | 是，包装为 JSON API |
| Tamper Lab | `compute_merkle_tree(request_json)` 返回完整树结构 | `@merkle.compute_root`（需扩展返回中间节点） | **需要底层扩展**：新增 `compute_tree` 返回所有中间哈希用于可视化 |
| Bench | 直接读 `report/_bench_data.csv` | 已有数据 | 无需后端 |

**关键底层扩展：** Tamper Lab 需要完整 Merkle 树结构（所有中间节点哈希），当前 `compute_root` 只返回根。需新增 `compute_tree(leaves) -> MerkleTree`，返回所有层级的节点。这是**唯一需要改底层**的项，且是纯增量（不破坏现有 API）。
